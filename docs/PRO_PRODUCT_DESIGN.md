# Pro Product Design

This document defines how Retrospective Coach Pro can capture the demand generated by the Free version.

## Product Positioning

Retrospective Coach Pro is not "a longer free prompt". It is a more complete, more stable, and more ready-to-use retrospective system.

It solves these problems:

- Users do not know how to make retrospectives more useful and concrete.
- Team retrospectives easily become status recaps, blame sessions, or vague summaries.
- Managers need to turn retrospective outputs into mechanisms, methods, and next-cycle actions.
- Consultants, trainers, and creators need reusable, explainable, deliverable retrospective assets.

## Target Users

| User | Core Need | Pro Value |
| --- | --- | --- |
| Personal growth users | Want to improve consistently, but retrospectives stay at the level of feelings | Personal retrospective templates, cycle retrospective templates, action tracking |
| Professionals | Need to review projects, reporting, interviews, and communication | Scenario templates, report-ready outputs, action lists |
| Managers | Need to facilitate team retrospectives and drive improvements | Meeting workflows, responsibility mechanisms, check signals |
| Consultants / trainers | Need deliverable method packages | Case library, teaching structure, assessment sheets |
| Enterprise customers | Need private deployment and customization | Deployment packages, industry templates, custom services |

## Pro Package

Recommended modules:

1. Pro main prompt
   - More complete role definition than Free.
   - Stronger evidence boundaries.
   - Better multi-turn questioning.
   - Better fit for complex team scenarios.

2. Multi-scenario templates
   - Project failure retrospective
   - Project success retrospective
   - Sales retrospective
   - Interview retrospective
   - Presentation retrospective
   - Content retrospective
   - Growth experiment retrospective
   - Meeting retrospective
   - Team phase retrospective
   - Personal monthly review
   - Manager decision retrospective
   - Customer delivery retrospective

3. Detailed workflow
   - Input preprocessing
   - Task clarification
   - Method selection
   - Analysis execution
   - Lesson extraction
   - Capability capture
   - Improvement action design
   - Report organization
   - Output quality check
   - Next-cycle tracking

4. Case library
   - At least two complete cases per scenario.
   - Includes user input, retrospective process, final output, and commentary.
   - Separates excellent output from common weak output.

5. Team implementation tools
   - Team retrospective meeting workflow
   - Pre-retrospective information collection form
   - Facilitator question checklist
   - Action item tracking sheet
   - Retrospective quality assessment sheet

6. Deployment package
   - GPTs configuration guide
   - Claude Project configuration guide
   - Coze agent configuration guide
   - Dify app configuration guide
   - Internal knowledge-base integration suggestions

## Free-to-Pro Conversion Path

The Free version should help users feel:

- A retrospective can be more than a summary.
- Good retrospectives separate facts from assumptions.
- Improvement suggestions should be concrete and checkable.

The Pro version captures deeper needs:

- I need more scenarios.
- I want my team to use this.
- I want to deploy it as an agent.
- I want case references and quality assessment.
- I want retrospectives to become an operating mechanism.

## Pro MVP Recommendation

Do not start too large. Recommended first Pro MVP:

1. Pro main prompt.
2. Six high-frequency scenario templates.
3. Six complete cases.
4. One team retrospective meeting workflow.
5. One action tracking sheet.
6. Deployment guides for GPTs, Coze, and Dify.

Expand more scenarios after user feedback.

## Recommended Launch Scenarios

Prioritize these six:

1. Project failure retrospective.
2. Project success retrospective.
3. Personal monthly review.
4. Interview retrospective.
5. Sales retrospective.
6. Team phase retrospective.

These scenarios have clear demand, frequent reuse, and strong demonstration value.

## Launch Rhythm

Suggested rhythm:

1. Week 1: publish the Free GitHub repository.
2. Week 2: collect 10-20 real feedback items.
3. Week 3: decide the six launch scenarios for the Pro MVP.
4. Week 4: open the Pro beta.
5. Week 5: collect beta cases and user testimonials.
6. Week 6: officially launch Pro.

## Risk Control

- Do not sell Pro as a "magic prompt"; sell it as a retrospective system.
- Do not promise enterprise capabilities too early unless delivery materials are ready.
- Do not put too many Pro-level templates into the Free version, or the paid boundary will blur.
- Do not only emphasize model capability; emphasize method, evidence, actions, and continuous improvement.
